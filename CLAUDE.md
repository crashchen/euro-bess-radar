# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

`euro-bess-radar` (a.k.a. eu-bess-pulse) is a local-first Streamlit dashboard for evaluating day-ahead arbitrage and merchant revenue potential of BESS across European bidding zones. Python 3.11+, pandas/numpy, SQLite cache, Plotly, openpyxl exports.

## Module Interaction (big picture)

`src/config.py` is the single source of truth for zones, timezones, cache paths, and revenue-math constants (`HOURS_PER_YEAR`, `ANCILLARY_CAPACITY_AVAILABILITY=0.95`, `ANCILLARY_ENERGY_ACTIVATION_SHARE=0.10`, `GBP_EUR_YEARLY`, `MAX_SHORT_GAP_HOURS=2.0`). Nothing else should hard-code zones, FX rates, or data-quality thresholds.

Data flow inside one dashboard run (`app.py`):
1. `data_ingestion.fetch_prices()` dispatches to the right backend (entsoe-py for ENTSO-E zones, Elexon REST for GB), runs `build_zone_query_window()` to convert local calendar dates to a UTC `[start, end)` window, and caches into `da_prices_{zone}` SQLite tables. Cache validity is checked per requested slice using row-level `fetched_at`, `_expected_cache_interval()` + `reindex`, so stale rows and gaps are detected as missing points instead of accepted as sparse data.
2. `analytics.py` consumes the UTC price frame but groups by the zone's local calendar day via the `tz` parameter. `_find_daily_ordered_trade()` (called from `calculate_daily_spreads()`) finds the best chronology-aware charge→discharge pair with rolling windows sized by `duration_hours`. Heatmaps, P50/P90, RE correlation, and revenue all share this ordered-spread basis.
3. `ancillary_fetchers.py` exposes a lazy `getattr()` registry (`fetch_{zone}_ancillary`) used by `run_auto_fetch()`. Results flow into `ancillary.normalize_auto_fetch_dataset()` and then `build_ancillary_dataset()`, which merges with any user-uploaded manual CSVs — manual uploads override auto-fetch *only for the same product type*, preserving other auto-fetched products.
4. `merge_revenue_stack()` combines DA arbitrage with per-product ancillary revenue and returns a `source_revenues` dict (e.g. `DA Arbitrage`, `FCR-N`, `aFRR Up`) that both the Revenue tab and `export.py` read from. Capacity-reserve ancillary is not treated as fully additive with DA in the headline total; `gross_additive_total_eur` remains available as a non-co-optimized reference, and `dispatch.solve_joint_capacity_batch()` can provide a screening-grade joint MILP power-headroom estimate.
5. `export.py` writes Excel reports that must respect the same `tz` parameter and stacked-revenue breakdown as the dashboard.

Tests in `tests/` are heavily mocked (no live API calls) and mirror the module layout 1:1. `conftest.py` holds shared price/ancillary fixtures.

## Data Sources

### ENTSO-E Transparency Platform
- Requires API key (loaded from .env)
- Covers: EU-27 + Norway (NO_1–NO_5) + Switzerland (CH)
- Python library: entsoe-py (>=0.7.7, uses `web-api.tp.entsoe.eu`; endpoint overridable via `ENTSOE_ENDPOINT_URL` env var)
- Returns: Day-ahead prices in EUR/MWh + generation by fuel type
- Resolution: 60min (most zones), 15min (DE_LU since Oct 2025)
- Historical data: available back to at least 2024 via the new API endpoint

### Elexon Insights API (Great Britain)
- NO API key required — fully public REST API
- Base URL: https://data.elexon.co.uk/bmrs/api/v1
- Endpoints:
  - GET /balancing/pricing/market-index — MID prices (GBP/MWh, 30-min)
  - GET /balancing/settlement/system-prices — system buy/sell prices (30-min)
  - GET /datasets/FUELINST — generation mix by fuel type
- GBP→EUR conversion uses year-specific rates via `GBP_EUR_YEARLY` dict in config.py
- Elexon returns multiple data providers per settlement period. Legitimate £0/MWh and negative prices must be retained; only recognizable duplicate zero placeholders are dropped before averaging remaining provider rows.

### Fingrid Open Data (Finland)
- API key recommended/required for v2 (`x-api-key` header via `FINGRID_API_KEY`)
- Base URL: https://data.fingrid.fi/api
- Dataset paths use `/datasets/{id}/data`
- Dataset IDs: 317 (FCR-N), 318 (FCR-D Up), 283 (FCR-D Down), 52 (aFRR Up), 51 (aFRR Down)
- Returns: hourly capacity prices in EUR/MW

### Regelleistung.net (Germany)
- NO API key required — public REST API
- Base URL: `https://www.regelleistung.net/apps/cpp-publisher/api/v1/download/tenders/resultsoverview`
- Params: `productTypes` (FCR/aFRR), `market` (CAPACITY), `exportFormat` (xlsx), `date` (YYYY-MM-DD)
- Auto-fetch enabled: downloads daily tender results as xlsx, parsed into standard ancillary format
- Manual CSV upload (`DE_FCR`, `DE_aFRR`) still supported as fallback and overrides auto-fetch for same product type
- Returns: capacity prices in EUR/MW

### Netztransparenz.de (Germany, reBAP/imbalance live fetch)
- Public German TSO transparency portal for balancing capacity/energy publications.
- Base URL in config: `NETZTRANSPARENZ_BASE_URL`.
- Implemented live fetch path: `data_ingestion.fetch_netztransparenz_imbalance()` calls the public `CsvDownloadHandler.ashx` chart endpoint for `NRV-Saldo qualitätsgesichert` (`ProduktId=6`) and `reBAP unterdeckt` (`ProduktId=10`, which carries both under/over symmetric reBAP columns), using the site's ME(S)Z/CET option so the returned `Zeitzone` column disambiguates DST.
- Writes through the same `persist_imbalance_frame(..., source="Netztransparenz.de")` cache/provenance path as manual reBAP/imbalance uploads; Data Trust and the cockpit overlay consume it unchanged.
- Not a replacement for Regelleistung.net tender-result capacity prices. Activated aFRR/mFRR energy datasets and GCC/LFC area balance remain manual/future until their schema and replay semantics are separately validated.

### ENTSO-E Imbalance Prices
- Uses same ENTSO-E API key as DA prices
- Covers: RO, SE_3, Italian bidding zones, and other ENTSO-E zones where imbalance data is available
- Returns: imbalance/balancing prices in EUR/MWh

### ENTSO-E Intraday Auction Prices (IDA1/2/3)
- Uses same ENTSO-E API key as DA prices via `entsoe-py.query_intraday_prices(zone, start, end, sequence)`.
- `INTRADAY_SUPPORTED_ZONES` (data_ingestion.py) lists the SIDC participants that publish auction results — currently `{DE_LU, NL, BE, FR, AT, IT_NORD}`. Other zones return `None` rather than raising.
- Sequence: 1 = IDA1 (15:00 D-1 opening auction), 2 = IDA2 (22:00 D-1), 3 = IDA3 (10:00 day-of-delivery). The dashboard currently uses IDA1 only.
- `entsoe-py` raises `NoMatchingDataError` (a `ValueError` subclass) when a supported zone has no IDA data for the requested window — this is mapped to `None`, not a hard error.
- Auth failures raise `DataSourceAuthError`; network failures raise `DataSourceNetworkError`.
- **Live IDA coverage is currently empty in practice**: ENTSO-E returns a `No matching data found` acknowledgement for the IDA auction query (`A44` + `contract_MarketAgreement.type=A07` + classificationSequence) across the tested SIDC zones/windows — an upstream data gap, NOT a code/parsing bug (verified at the raw-HTTP layer). The `diag/ida-resolution` branch holds the reproducible diagnostic.
- **Manual IDA CSV fallback**: `parse_intraday_csv()` + `persist_intraday_frame()` (data_ingestion.py) let users upload IDA prices when the API has nothing. Minimal schema `timestamp, ida_price_eur_mwh` with optional per-row `sequence` and `zone`; rows write to the SAME `ida_prices_{zone}_seq{n}` SQLite tables the live fetch uses, so the Revenue/Cockpit cache-first reads consume them unchanged. `app.py` reads the IDA cache cache-first so the cockpit sees uploads without a Revenue-tab visit. `sequence` accepts `1/2/3` or `IDA1/IDA2/IDA3` labels (case-insensitive) and falls back to the UI default round only when blank — fractional (`1.5`) or non-numeric (`bogus`) values RAISE rather than silently coercing to the default round (which would file IDA2/IDA3 rows under the IDA1 table); validation runs only on rows surviving the timestamp/price dropna.
- **IDA provenance is durable AND per-row**: `write_intraday_cache(..., source=...)` stores the source on every price row (migrating pre-provenance tables by backfilling a legacy sentinel) and upserts a derived label into the `ida_price_sources` sidecar — uniform source ⇒ that label, multiple sources in one table ⇒ `Mixed`. Data Trust's `build_intraday_source_table()` reads that sidecar (via `read_intraday_sources()`), so a `(zone, sequence)` mixing manual + live intervals shows `Mixed` rather than only the last write's source, and labels survive a session restart.

## Ancillary Services

### Auto-Fetch (ancillary_fetchers.py)
Zone-specific fetchers run automatically when the zone is selected:
- **FI** — FCR-N/D + aFRR prices via Fingrid
- **DE_LU** — FCR/aFRR auction results via Regelleistung.net REST API (downloads daily xlsx)
- **GB** — System buy/sell prices via Elexon
- **RO, SE_3, IT_NORD, IT_CNOR, IT_CSUD, IT_SUD, IT_CALA, IT_SICI, IT_SARD** — Imbalance prices via ENTSO-E

Registry uses lazy `getattr()` lookup for testability. `run_auto_fetch()` handles per-fetcher errors gracefully.

### Manual Upload (CSV Templates)
Supported templates:
- **DE_FCR** — Germany FCR from regelleistung.net
- **DE_aFRR** — Germany aFRR from regelleistung.net
- **RO_BALANCING** — Romania balancing from Transelectrica
- **FI_FCR** — Finland FCR from Fingrid
- **GB_BALANCING** — GB system prices from Elexon/BMRS

Upload via sidebar. Template CSVs downloadable from the UI. A separate **Intraday (IDA) Prices** sidebar uploader handles IDA1/2/3 price CSVs (see the ENTSO-E Intraday section) — it does not use the `ANCILLARY_TEMPLATES` registry.

**Unified reserve-capacity import (Step 2, import-first)**: `ancillary.generate_capacity_import_template_csv()` is the single zone-tagged capacity format — columns `timestamp, zone, product, direction, capacity_price_eur_mw_h` (+ optional `timezone`), downloadable from the sidebar and documented in `docs/import-templates.md` (the spec to hand to an exchange/TSO). Two semantics are pinned in the template header: timestamps are UTC (or carry a `timezone` column), and `capacity_price_eur_mw_h` is a PER-HOUR rate, never a 4h-block total. The unified parser `ancillary.parse_capacity_import_csv()` (increment 6b) ingests this into the standard ancillary frame: `product`→`product_type` canonicalised to FCR/aFRR/mFRR and `direction`→up/down/symmetric (case/separator tolerant; an unrecognised value RAISES, like the IDA importer), per-row `timezone` honoured (else UTC), and `capacity_price_eur_mw_h` mapped to the existing `capacity_price_eur_mw` column (same unit, no codebase-wide rename). `product_type`/`direction` are kept faithful to the CSV (direction is NOT folded into the product label). Persistence (also 6b): `data_ingestion.persist_capacity_frame()` / `write_capacity_cache()` write one `capacity_prices_{zone}` table keyed `(timestamp, product, direction)` (keep-last on re-import via `INSERT OR REPLACE`), with a `capacity_price_sources` provenance sidecar keyed `(zone, product, direction)` — full parity with the IDA `ida_price_sources` path. The sidecar label is RE-DERIVED per stream after every write (uniform source ⇒ that label, multiple ⇒ `Mixed`, `read_capacity_sources()`), so an overwrite that changes the source refreshes provenance rather than leaving a stale label on fresh data. `read_capacity_cache()` reads it back as a standard frame. Data Trust (6c slice 1) renders a capacity-source table and shows reserve PER zone in the coverage matrix from this sidecar (see the Data Trust bullet). A sidebar uploader (6c slice 2, `sidebar._parse_and_store_capacity_upload`, in the **Ancillary Services Data** expander under **Unified Reserve Capacity CSV**) calls `parse_capacity_import_csv` + `persist_capacity_frame(source="Manual CSV")`, shows a per-(zone,product,direction) import summary, and surfaces `DataSourceParseError` as a friendly message — it coexists with (does NOT replace) the per-country session-ancillary uploader. Cockpit consumption (6c slice 3) closes the loop: the forecast-driven IDA-policy panel resolves capacity **cache-first** via `_resolve_capacity_dataset(primary_zone, anc_df)` — `read_capacity_cache(primary_zone)` (imported unified capacity) drives the 9.2b reserve rows, falling back to the session ancillary only when the cache is empty, with a "Reserve capacity source: Cached unified import / Session ancillary fallback" caption. The resolved full-zone frame is windowed per valid date via `_slice_to_local_dates` before pricing, so a full-sample mean never leaks into the comparison window. **Step 2 import-first loop is now closed end-to-end: template → upload → parse → SQLite + provenance → Data Trust → cockpit 9.2b model.**

**Activation-energy import (Step 3, import-first, started 3a):** the ENERGY leg of reserves — paid only when the TSO actually calls the reserve, distinct from the standing capacity fee. Following the same import-first playbook (template/spec → parser → Data Trust → cockpit replay; activation/GCC live fetchers only after their schema/semantics are stable, NOT first). **3a (template + spec only, no parser):** `ancillary.generate_activation_import_template_csv()` (constants `ACTIVATION_IMPORT_COLUMNS/PRODUCTS/DIRECTIONS`) — columns `timestamp, zone, product, direction, activation_price_eur_mwh, system_activated_volume_mw` (+ optional `timezone`); `product` is aFRR/mFRR (FCR has no separately-paid energy leg), `direction` is up/down (energy activation is directional, no symmetric). Downloadable from the sidebar (Ancillary Services Data expander) and specced in `docs/import-templates.md` §3. **Three red-lines pinned in the template header + docs (lock BEFORE the parser):** (1) `system_activated_volume_mw` is SYSTEM/area-level activated power, NOT this asset's output — the asset/capture share is a model assumption that lives in the audit panel, never pre-mixed into the CSV; (2) activation energy is a SEPARATE stream from the capacity fee AND from reBAP/imbalance settlement and is NOT free additive revenue (activating up spends SoC DA/IDA could have sold — it competes for the same energy budget); (3) with no forward activation signal it supports HISTORICAL REPLAY ONLY, never live dispatch or aggregator activation-following. **3b LANDED (parser + persistence + provenance, full parity with the capacity 6b path):** `ancillary.parse_activation_import_csv()` returns a DEDICATED activation frame (timestamp-indexed UTC; `zone, product_type, direction, activation_price_eur_mwh, system_activated_volume_mw`) — NOT the standard ancillary frame, because it carries energy-leg columns the standard frame lacks AND because activation energy is a separate stream. `product` canonicalised to aFRR/mFRR (**FCR RAISES** — no paid energy leg), `direction` to up/down (**`symmetric` and blank RAISE** — energy is directional, unlike capacity where blank⇒symmetric); unknown product/direction RAISE (IDA/capacity strictness); bad timestamp/price/volume rows DROP. `system_activated_volume_mw` is stored SYSTEM-level exactly as parsed — no asset/capture-share multiplication anywhere in the parser or persistence (the red-line). Persistence: `data_ingestion.persist_activation_frame()`/`write_activation_cache()` write one `activation_prices_{zone}` table keyed `(timestamp, product, direction)` (keep-last via `INSERT OR REPLACE`) + an `activation_price_sources` sidecar keyed `(zone, product, direction)` with the label re-derived per stream after each write (uniform⇒source, multiple⇒`Mixed`, else `Legacy`; `read_activation_sources()`), so an overwrite refreshes provenance. `read_activation_cache()` reads it back. Sidebar uploader (`sidebar._parse_and_store_activation_upload`, "Unified Activation-Energy CSV" under Ancillary Services Data) wired to the existing template download. The shared timestamp helper `ancillary._import_utc_index` (renamed from `_capacity_utc_index`) now serves both importers. **Zone-name safety (Gemini security review, PR #18):** raw CSV `zone` values flow into per-zone SQLite table names (`activation_prices_{zone}` / `capacity_prices_{zone}`), so `data_ingestion.validate_import_zone()` enforces table-name safety (`re.fullmatch(r"[A-Za-z0-9_]+")`, non-empty) at BOTH the parser layer (`ancillary._resolve_import_zone_cell`, early friendly error — also applied to the capacity parser) and the `_{activation,capacity}_cache_table()` choke point (guards a caller bypassing the parser). It is deliberately NOT the supported-bidding-zone check (`_validate_zone`) — import-first may ingest a zone not yet wired into config; this is table-name safety, not business support. **3c-1 LANDED (Data Trust only):** `data_trust.build_activation_source_table()` (reads `read_activation_sources()`) renders an "Activation-energy price sources" table at `(zone, product, direction)` granularity, and `build_coverage_matrix()` gains a PER-zone `activation_energy` column — import-only, so NO session fallback (unlike the reserve column). No model/revenue calc. **3c-2a LANDED (replay-calc only, no UI):** `activation_overlay.compute_activation_overlay(activation_df, *, reserve_mw, capture_share, interval_hours=None)` → `{activation_energy_overlay_eur, by_stream[(product,direction) → activated_mwh, activation_overlay_eur], reserve_mw, capture_share}`. Formula: `asset_activated_mwh = min(reserve_mw, capture_share · max(system_activated_volume_mw, 0)) · dt`; `cashflow = asset_activated_mwh · activation_price_eur_mwh`. **Locked semantics:** the activation price is NOT sign-flipped by direction (it is a cash-flow price already; a down activation at a negative price stays negative — no double-sign error); `system_activated_volume_mw` is floored non-negative (a negative is never a sign); `capture_share` is a global scalar (≥0, not auto-clamped above 1 — the `reserve_mw` cap bounds delivered power); `dt` inferred per stream from timestamp spacing (overridable). Direction is a breakdown label + future-SoC hook only. Pure calculation: NO UI, NO SoC consumption, NO MILP, NOT additive to the strategy total (the headline key is `activation_energy_overlay_eur`, deliberately not "revenue"). **3c-2b LANDED (cockpit panel):** `simulation_cockpit._render_activation_overlay_section` (gated on `read_activation_cache(primary_zone)` having data) — a capture-share slider (percent units, default 1%, range 0–10%; `app.py` converts to a fraction for the audit table), windows the activation cache to the loaded dates via `_slice_to_local_dates`, calls `compute_activation_overlay`, and shows the `activation_energy_overlay_eur` metric + per-(product,direction) breakdown under a hard caption (replay overlay; NOT co-optimized, NOT additive to the strategy total, no SoC coupling, not aggregator dispatch; assumes a regular interval series). `build_assumptions_table(activation_capture_share=…)` appends three audit rows (capture share / regular-interval interpretation / non-additive overlay basis). `interval_hours` is deliberately NOT exposed in the UI (it would imply the panel can fix dirty data); the regular-interval requirement is documented in `docs/import-templates.md` §3 instead. **Activation-energy import→model loop is now CLOSED end-to-end: template → upload → parse → SQLite + provenance → Data Trust → cockpit overlay.** Activation/GCC live fetchers remain future/manual until those source schemas are validated; the implemented Netztransparenz live path today is the separate reBAP/NRV imbalance fetcher in Step 4f.

**reBAP / imbalance settlement import (Step 4, import-first, 4a–4f):** the passive BRP/portfolio imbalance-settlement stream, distinct from reserve-capacity fees and activation-energy cashflows. **4a LANDED (template + spec):** `ancillary.generate_imbalance_import_template_csv()` (constant `IMBALANCE_IMPORT_COLUMNS`) — columns `timestamp, zone, imbalance_price_eur_mwh, system_imbalance_volume_mw` (+ optional `timezone`, documented in `docs/import-templates.md` §4) and downloadable from the sidebar (Ancillary Services Data expander). **Four red-lines pinned before model work:** (1) this is a separate passive settlement strategy, not capacity fee and not activation energy, so do NOT blindly add it to reserve revenue; (2) `system_imbalance_volume_mw` is SYSTEM/area imbalance, NOT this asset's imbalance — asset imbalance/share assumptions belong in the audit panel later; (3) `imbalance_price_eur_mwh` is the published cash-flow settlement price (negative values valid), with no extra direction sign flip; (4) historical replay/diagnostics only, not live BRP control or aggregator dispatch. **4b LANDED (parser + persistence + provenance + sidebar uploader):** `ancillary.parse_imbalance_import_csv()` returns a DEDICATED imbalance frame (timestamp-indexed UTC; `zone, imbalance_price_eur_mwh, system_imbalance_volume_mw`) — NOT the standard ancillary frame, because imbalance settlement is a separate passive stream. Bad timestamp/price/volume rows DROP; missing required columns or unsafe zones RAISE. `system_imbalance_volume_mw` and signed `imbalance_price_eur_mwh` are stored exactly as parsed — no asset-share multiplication and no direction sign flip anywhere in parser/persistence. Persistence: `data_ingestion.persist_imbalance_frame()`/`write_imbalance_cache()` write one `imbalance_prices_{zone}` table keyed by `timestamp` (keep-last via `INSERT OR REPLACE`) + an `imbalance_price_sources` sidecar keyed by `zone` with the label re-derived after each write (uniform⇒source, multiple⇒`Mixed`, else `Legacy`; `read_imbalance_sources()`), so an overwrite refreshes provenance. `read_imbalance_cache()` reads it back. Sidebar uploader (`sidebar._parse_and_store_imbalance_upload`, "Unified reBAP / Imbalance CSV" under Ancillary Services Data) is wired to the existing template download. Zone-name safety uses the shared `data_ingestion.validate_import_zone()` at parser and table-name choke-point. **4c LANDED (Data Trust only):** `data_trust.build_imbalance_source_table()` renders a "reBAP / imbalance settlement sources" table keyed by zone, and `build_coverage_matrix()` gains a PER-zone `imbalance_settlement` column showing `source (rows)` from `read_imbalance_sources()`; import-only, so NO session fallback. **4d-1 LANDED (replay-calc only, no UI):** `imbalance_overlay.compute_imbalance_overlay(imbalance_df, *, power_mw, capture_share, interval_hours=None)` → `{imbalance_settlement_overlay_eur, by_system_state[(system_short/system_long/neutral) → asset_imbalance_mwh, imbalance_overlay_eur], power_mw, capture_share, sign_convention}`. Sign convention was validated against German Netztransparenz May-2026 NRV-Saldo + reBAP CSVs and the official NRV-Saldo-Ampel wording: `system_imbalance_volume_mw > 0` = system short / undercovered, so positive BESS net dispatch (discharge / extra injection) helps; `< 0` = system long / overcovered, so negative BESS net dispatch (charge / less injection) helps. Formula: `asset_net_dispatch_mw = sign(system_imbalance_volume_mw) * min(power_mw, capture_share * abs(system_imbalance_volume_mw))`; `cashflow = asset_net_dispatch_mw * dt * imbalance_price_eur_mwh`. `imbalance_price_eur_mwh` remains the published signed cash-flow price — no second sign flip — so helping the system can still lose money when the price sign is unfavourable. Pure calculation basis: no SoC/energy sustainability, no MILP, and not additive to the strategy total (headline key is `imbalance_settlement_overlay_eur`, deliberately not "revenue"). **4d-2 LANDED (cockpit panel):** `simulation_cockpit._render_imbalance_overlay_section` (gated on `read_imbalance_cache(primary_zone)` having data) windows the imbalance cache to the loaded dates via `_slice_to_local_dates`, exposes a capture-share slider (percent units, default 1%, range 0-10%; `app.py` converts to a fraction for the audit table only when cached imbalance rows overlap the current zone/window, preventing Streamlit session-state leakage across zones), and shows the signed `imbalance_settlement_overlay_eur` metric + by-system-state breakdown. Hard caption: historical replay overlay only, NOT co-optimized, NOT additive to the strategy total, ignores SoC/energy sustainability (more optimistic than activation because passive imbalance can be sustained), and not live BRP control or aggregator dispatch. `build_assumptions_table(imbalance_capture_share=...)` appends four audit rows (capture share / sign convention / non-additive overlay basis / power cap). **Imbalance import->model loop is now CLOSED end-to-end: template -> upload -> parse -> SQLite + provenance -> Data Trust -> cockpit overlay.** **Manual Netztransparenz converter (4e):** `scripts/convert_netztransparenz_imbalance.py` converts official German `NRV-Saldo qualitaetsgesichert` + `reBAP unterdeckt/ueberdeckt` CSV exports into the unified imbalance template (`timestamp,zone,imbalance_price_eur_mwh,system_imbalance_volume_mw`). It is a thin CLI wrapper over `data_ingestion.convert_netztransparenz_imbalance_exports` (public API) — the SAME converter the 4f live fetch uses — so manual and live paths share one set of semantics: German comma-decimal normalisation, official `N.A.` unavailable intervals dropped (after the raw-axis 15-minute regularity check), NRV/reBAP inner join (only timestamps published in both survive; fully disjoint files raise), CEST/CET DST disambiguation via the `Zeitzone` column, symmetric under/over reBAP tolerance on published rows, and a final round-trip of the generated CSV through `parse_imbalance_import_csv`. See `docs/runbooks/validate-imbalance-overlay.md` for the May-2026 real-data smoke test. **Live Netztransparenz fetcher (4f):** `data_ingestion.fetch_netztransparenz_imbalance("DE_LU", start, end)` calls the public `CsvDownloadHandler.ashx` chart endpoint directly (base64-encoded `{LocalFrom, LocalTo, ResultTimeZone="cet", Settings}` payload), downloads `NRV-Saldo qualitätsgesichert` (`ProduktId=6`) + `reBAP unterdeckt` (`ProduktId=10`, includes both under/over columns), validates the raw response timestamp axes and German DST labels, explicitly parses German comma-decimal numbers, drops official unavailable value tokens such as `N.A.` before taking the NRV/reBAP inner join, validates reBAP under/over equality on published rows, returns the dedicated imbalance frame, and the sidebar button **Fetch Netztransparenz reBAP/imbalance** persists it with source `Netztransparenz.de`. Manual upload and live fetch therefore share one cache/provenance/Data Trust/cockpit path.

### Data Merging
`build_ancillary_dataset()` merges manual + auto-fetch data. Manual uploads override auto-fetch for the same product type only — other auto-fetched products are preserved. `normalize_auto_fetch_dataset()` converts varied fetcher schemas into a standard ancillary format with per-product rows. Override matching uses `_normalize_product_key()` (case- and separator-insensitive) so an upstream label flip from `aFRR Up` to `afrr_up` does not silently bypass the override.

### UI Guidance
- Revenue Estimation tab shows an annualisation note based on the currently selected sample window length
- Revenue Estimation tab includes a `How ancillary works` expander explaining product-level stacking, manual-vs-auto precedence, and which ancillary signals are not auto-monetised

### Auth Errors
Fingrid / Regelleistung / Elexon all detect HTTP 401/403 via the shared `_raise_if_auth_failed()` helper and raise `DataSourceAuthError` *before* the `@retry` decorator can swallow it. `run_auto_fetch()` deliberately does not catch `DataSourceAuthError`, so it propagates to `_run_and_store_ancillary_fetch()` in `src/components/sidebar.py` which surfaces it via `st.error` with a hint (e.g. "Set FINGRID_API_KEY in .env."). Other transient fetcher errors (network, parse) are still caught per-fetcher and logged.

### Data Quality
`clean_prices()` short-gap interpolates gaps up to `MAX_SHORT_GAP_HOURS` (config.py) and flags those rows with `imputed=True`. Longer gaps stay NaN with `filled=True / imputed=False`. `_segment_and_reindex_prices()` also detects sparse internal gaps when no `expected_window` is provided (mode-delta heuristic). For zones listed in `_ZONE_RESOLUTION_TRANSITIONS` (declare new ones there when a TSO changes its market time unit) the helper splits the index at each transition boundary and runs the mode-delta heuristic on each side independently — this surfaces sparse gaps inside both the pre- and post-transition segments without upsampling the lower-resolution side. `analytics.filter_to_complete_local_days(df, tz)` mirrors the day-level filter that `calculate_daily_spreads()` uses, and is applied in `app.py` and `src/export.py` so negative-price stats and heatmaps stay consistent with daily-spread / dispatch / revenue analytics across the dashboard, Excel, and PDF outputs. Excel/PDF exports always recompute the negative-price stats from the filtered subset internally; do not rely on the caller-supplied `negative_stats` to be already filtered.

## Coding Conventions
- pandas timestamps are UTC internally; every analytics/export entry point takes a `tz` parameter for local-time grouping. Do not group by UTC day — daily spread and heatmap numbers will be wrong across DST boundaries.
- Add new zones in `config.py` only: `ENTSOE_ZONES`/`ELEXON_ZONES` + `ZONE_TIMEZONES`. Never hard-code zone codes elsewhere.
- SQLite table naming is `da_prices_{zone_code}` lowercase (e.g. `da_prices_de_lu`).
- GBP→EUR conversion must use `_get_gbp_eur_rate_for_year()` (nearest-year fallback against `GBP_EUR_YEARLY`), never a single scalar rate — GB history spans multiple FX regimes.
- API calls retry with exponential backoff, max 3 retries.
- Keep functions under ~50 lines; split long pipelines at natural stage boundaries.
- Logging via `logging`, Google-style docstrings, type hints on public functions.
- Streamlit ≥1.49 API: use `width="stretch"` for `st.dataframe` / `st.plotly_chart`, NOT `use_container_width=True` (deprecated). The repo targets 1.55.0; automated reviewers trained on older Streamlit may flag `width="stretch"` as invalid — it is correct, do not "fix" it back.

## Key Domain Knowledge
- **Ordered spreads**: Revenue estimation uses chronology-aware charge-before-discharge windows, not simple max-min. `_find_daily_ordered_trade()` finds best non-overlapping buy/sell pair using rolling averages and backward scan.
- **Duration-aware analysis**: `duration_hours` parameter (1h/2h/4h) controls rolling window size for spread calculation, heatmaps, and zone comparison. Models actual BESS dispatch constraints.
- **Zone-local time**: All analytics group by local calendar day/hour (via `tz` parameter), not UTC. Critical for correct daily spread, heatmap, and export date ranges.
- Germany (DE_LU) switched to 15-min DA resolution in Oct 2025
- GB data is 30-min settlement periods, GBP/MWh (not EUR)
- **Sub-hourly resolution**: `calculate_negative_price_hours()` returns both `negative_hours` (physical hours) and `negative_intervals` (count of data points) to handle 15-min/30-min data correctly
- Norway and Switzerland are NOT EU members but their TSOs are ENTSO-E members — their data is on the Transparency Platform
- Some ENTSO-E zones have data gaps — always validate returned DataFrame is not empty
- DA prices across ENTSO-E are in EUR/MWh; GB is in GBP/MWh
- Generation data: not all zones have solar/wind/offshore split — handle missing columns gracefully
- Negative wind/solar-price correlation = BESS-friendly market (high RE → low prices = charging, low RE → high prices = discharging)
- **Revenue stacking**: `merge_revenue_stack()` exposes DA arbitrage + ancillary `source_revenues` for per-product breakdown, but capacity-reserve ancillary is not assumed to be fully additive with DA in the headline total without co-optimization.
- **Joint co-optimization**: `solve_joint_capacity_batch()` models reserve capacity as power headroom competing with DA charge/discharge in a MILP with binary charge/discharge mutual exclusion. It does not model activation energy, bid acceptance, reserve-specific SoC duration, or qualification constraints.
- **Mutual exclusion is binary, not relaxed**: `solve_daily_lp()` and `solve_daily_joint_capacity_lp()` both use a binary mode variable via scipy `integrality`. A pure LP relaxation of `p_charge + p_discharge <= power_mw` is degenerate at negative prices (the solver can earn revenue by simultaneously charging and discharging through round-trip losses). Do not weaken this back to an LP-only constraint.
- **Annualisation caveat**: DA arbitrage revenue is extrapolated from the user-selected sample window, so short windows (for example winter-only periods) can materially overstate or understate full-year merchant potential
- **Portfolio analysis** (`src/portfolio.py`): treats each zone as a daily-revenue-per-MW series and runs Pearson correlation, Sharpe-like mean/std, and a long-only Markowitz frontier via scipy SLSQP. `build_daily_revenue_matrix()` inner-joins on dates so a single missing local day in any zone drops that date from the whole portfolio view — this keeps correlations and frontier weights computed on a single common sample. Annualisation uses the same `365.25`-day convention as `estimate_annual_arbitrage_revenue` and assumes i.i.d. days (`std_annual = std_daily * sqrt(365.25)`); for short or seasonal samples treat the Sharpe number as a relative ranking, not an absolute investment metric.
- **Forward scenarios** (`src/forward_curve.py`): user-supplied forward-curve CSV (zone, delivery_start, delivery_end, price_eur_mwh, optional contract/shape/source/as_of). `build_forward_synthetic_prices` overlays each contract's baseload onto the historical hour-of-week shape for the same zone: `hourly_price[t] = forward_base * historical_shape[hour_of_week(t)]`. The synthetic series is then fed through the existing daily-spread / MILP-dispatch / annualisation pipeline, so the dashboard reuses one consistent revenue model across historical and forward views. Overlapping contracts (same zone, e.g. a Cal-2027 + Jan-2027 quote) keep the later one in CSV order on the overlap region; the UI also lists the overlap rows in a warning so users can resolve the intent themselves. Shape recovery requires the zone's historical DA series to be loaded in the sidebar first.
- **Data Trust tab** (`src/data_trust.py`, `src/pages/data_trust.py`): displays one row per fetched zone with source label, timezone, first/last UTC timestamp, coverage %, source gaps, short-gap imputations, unresolved missing intervals, and max gap hours. Keep this as the user-facing audit label for screening outputs whenever data ingestion or cleaning semantics change. It also renders a **coverage matrix** (`build_coverage_matrix()`) — one row per touched zone (loaded DA zones + zones with cached IDA provenance + the primary zone) with a short status per stream (`DA` coverage%, `IDA1/2/3` `source (rows)`, `Reserve (capacity)` product list, `Activation (energy)` product list, `Imbalance (reBAP)` `source (rows)`, or `—`) so a user sees at a glance what is loaded before trusting any number. The reserve column is shown PER zone from the persisted `capacity_price_sources` sidecar (the 6b unified-import path; `read_capacity_sources()`), falling back to the session ancillary's `primary_zone`-only products when a zone has no persisted unified-import capacity; `app.py` passes `anc_df` + `primary_zone` into `data_trust.render`. The `activation_energy` column (3c-1) is likewise per-zone from the `activation_price_sources` sidecar (`read_activation_sources()`) but import-only (no session fallback). The `imbalance_settlement` column (4c) is per-zone from the `imbalance_price_sources` sidecar (`read_imbalance_sources()`), also import-only with no session fallback. When IDA prices are manually uploaded, `build_intraday_source_table()` adds a dedicated sub-table labelling each `(zone, sequence)` as `Manual CSV` so IDA-driven cockpit/uplift numbers stay traceable, and `build_capacity_source_table()` does the same for reserve capacity at `(zone, product, direction)` granularity (reading the same `capacity_price_sources` sidecar), with `build_activation_source_table()` doing likewise for activation energy from the `activation_price_sources` sidecar and `build_imbalance_source_table()` doing likewise for passive reBAP/imbalance settlement from the `imbalance_price_sources` sidecar. It also renders a **Model assumptions & haircuts** table from `assumptions.build_assumptions_table()` (power/duration/efficiency/capture/capex/dispatch-model from the sidebar; VOM, 365.25-day annualisation, ancillary availability/activation shares, short-gap cap, replay interval cap from config; and the interactive panel knobs — rebid deadband, IDA forecast mode/bucket, rebid share — when set) so every screening number's provenance is answerable in one place. `app.py` builds this table once before the tabs and shares it with both Data Trust and the cockpit export.
- **Simulation Cockpit** (`src/simulation.py`, `src/pages/simulation_cockpit.py`): interval-level replay of the existing MILP dispatch model for a selected local day, plus dispatch event tables and multi-day replay summaries. It reports SoC, physical dispatch, cumulative revenue, throughput/FCE, trade blocks, C-rate, DoD, and screening-level degradation. Treat it as a historical/forward replay cockpit, not live telemetry or real trading execution. Reserve headroom overlay is physical uncommitted headroom only; do not present it as FCR/aFRR activation power without activation data. The multi-day replay and forecast-policy panels each offer an Excel download via `export.cockpit_tables_to_excel()`, which bundles the shown result table plus the model-assumption audit as an `Assumptions` sheet so an exported file is self-documenting. The forecast-policy panel also renders a **strategy comparison** (`strategy_compare.build_strategy_comparison()`) that reframes the sequential batch's three benchmarks (DA-only / forecast-driven DA+ID / perfect-foresight ceiling) as an investment view — window revenue, annualised EUR/MW/yr (365.25 i.i.d. convention), and uplift% vs DA-only — as a grouped bar chart + table, also included in that panel's export. An **optional fourth row** appears when ancillary capacity prices are loaded: a **DA + reserve-capacity co-optimisation (headroom)** strategy, computed by summing `dispatch.solve_joint_capacity_batch()`'s `joint_total_revenue` over EXACTLY the DA+ID rows' valid days (`per_day["date"]`, via `_reserve_coopt_total()`, so the annualisation denominator matches across all rows). The capacity price itself is also taken from ancillary rows INSIDE that same valid-date window (`_slice_to_local_dates()` before `capacity_price_for_product()`), so an out-of-window capacity print cannot skew the row; when no capacity rows overlap the window the row is omitted (with a UI note) rather than priced off the full sample. This is a **headroom-aware capacity estimate, NOT activation-energy revenue, and NOT additive with DA** — the joint MILP makes reserve headroom compete with DA charge/discharge for the same power budget, so the red-line is enforced by the solver, not by labelling (`ANCILLARY_CAPACITY_AVAILABILITY=0.95` haircut applies; activation energy is not modelled). It is a *different revenue stream* from the DA+IDA rows, **not** a cumulative DA+IDA+reserve ladder; the cockpit caption (`has_reserve` branch) and the export's reserve assumption rows (`_append_reserve_assumptions()`) state this explicitly. The reserve price is `ancillary.capacity_price_for_product()` (duration-weighted mean) for a product chosen from `ancillary.list_capacity_products()`; only DE_LU/FI (or manual CSV) carry capacity prices today, so the row is simply absent for other zones. A **fifth row** — **DA + IDA1 + reserve (co-opt ceiling)** — is the cumulative perfect-foresight ladder step (Phase 9.2a): `simulation.simulate_da_id_reserve_ceiling_batch()` sums `dispatch.solve_daily_da_id_reserve_dispatch()`, which **introduces no new MILP formulation** (it still calls the existing joint-capacity MILP) — it fuses the DA+ID ceiling accounting (`total = da_gross − implicit_mtm + stage_2_total`, DA financial and marked-to-market at IDA) with Stage 2 = `solve_daily_joint_capacity_lp()` at IDA prices. So reserve headroom competes ONLY with the *physical* Stage-2 schedule (`stage2_charge + stage2_discharge + reserve ≤ power_mw`); the DA financial commitment does NOT occupy physical headroom, and IDA "rebid" is cash settlement of the DA-vs-physical difference, not an extra physical power variable. It collapses to the DA+ID ceiling at `capacity_price=0` and to the DA+reserve row when `IDA==DA`, and satisfies triple ≥ DA+ID ceiling ≥ DA-only (regression-tested). It shares the v1 row's in-window capacity price and valid-day window, needs both in-window IDA and capacity prices, keeps the same red-line (headroom only, no activation energy), and is a **perfect-foresight upper bound, NOT forecast-driven** (`_append_triple_assumptions()` states this in the export). A **sixth row** — **DA + IDA1 + reserve (forecast-driven realistic)** — is the Phase 9.2b reserve-first sequential policy (`simulation.simulate_sequential_da_id_reserve_batch`), ALWAYS walk-forward (reserve is committed D-1 before the DA gate and must not see the target/future day, independent of the panel's LOO/walk-forward toggle, which only scores the DA/IDA skill estimate). The cockpit prices BOTH this row and the 9.2a ceiling (5th row) off the SAME per-interval reserve series (`ancillary.capacity_price_series_for_product()` windowed to the DA+ID valid dates) via `_reserve_triple_totals()`, so the displayed ceiling equals the 9.2b batch's `total_global_ceiling_eur` and **`realistic ≤ ceiling` by construction**; both rows are scored over the 9.2b walk-forward window (their own `triple_valid_days`/`triple_da_baseline` in `build_strategy_comparison()`, which may span one fewer day than the DA/IDA rows). A **forecast-effect gap panel** (`_render_reserve_gap_panel()`) decomposes the full gap (ceiling − realistic) into a SIGNED **Forecast effect** (negative = the forecast helped) plus a commitment-timing cost (`full_gap == forecast_effect + timing_cost`); same red-line, stated in the export via `_append_realistic_triple_assumptions()` (reserve-first, walk-forward, capacity headroom only, no activation energy). A separate **activation-energy overlay** panel (`_render_activation_overlay_section`, 3c-2b; gated on `read_activation_cache`) shows `activation_overlay.compute_activation_overlay()`'s `activation_energy_overlay_eur` + per-(product,direction) breakdown under a capture-share knob, explicitly as a historical replay overlay — NOT co-optimized, NOT additive to the strategy total, no SoC coupling (see the Activation-energy import paragraph). A scenario-based stochastic MILP remains the heavier future upgrade.
- **Multi-day SoC carry-over** (`simulate_replay_batch`, DA-only and DA+ID): `carry_soc=True` (default) routes to `_simulate_continuous_da_replay` (DA-only) or `_simulate_continuous_da_id_replay` (DA + IDA1), which group the requested local dates into maximal contiguous clean runs and solve each run as ONE solver call — terminal-neutral equality applies only at the END of the run, so day N can end with stored SoC that day N+1 monetises. The DA+ID path reuses the horizon-agnostic `solve_daily_da_id_dispatch` (it runs `solve_daily_lp` on the run's DA prices and again on its IDA prices, both terminal-neutral at the run end), so SoC carries across days in both the committed DA stage and the ex-post IDA rebid stage. Per-day rows are reconstructed by slicing the run's charge/discharge/soc arrays at day boundaries; `soc_start_pct` / `soc_end_pct` are real per-day boundaries within the run, not 50% resets. Because the run is solved globally, an individual day's revenue can be *lower* than its standalone single-day solve (e.g. it ends at 100% SoC to feed the next morning's spike), but the run total is always ≥ the sum of independent single-day solves. `batch.attrs["carry_mode"] = "continuous_horizon"` labels this regime in the UI; runs are broken by a missing day, an empty slice, any NaN value, or a **sparse day** (`_is_regular_utc_day` rejects a day whose UTC index has a non-uniform interval delta — e.g. a missing 02:00 the upstream cleaner did not reindex to NaN, or a DA/IDA inner-join that dropped intervals — which would otherwise let the single-`dt` continuous MILP silently compress the time axis). The check is DST-safe: spring-forward (23 local hours) and fall-back (25 local hours) days stay uniform in UTC and are kept. Run grouping is shared via `_group_clean_runs(dates, build_day=...)` where `build_day` yields either the DA-only slice or the merged DA+IDA frame. Each run boundary acts as a soft reset and the next run starts at the end-SoC of the previous run. **Performance cap**: a contiguous run is also split when adding the next day would exceed `config.MAX_CONTINUOUS_REPLAY_INTERVALS` (default 576), because a single binary MILP over a long horizon scales superlinearly — a 90-day hourly solve already takes ~40s and a 90-day 15-min run (8640 intervals) effectively hangs the dashboard. The cap is by *interval count*, not days, so it auto-adapts to resolution (576 ≈ 24 days hourly or 6 days at 15-min); the 15-min market time unit is the binding case (its MILP is ~7× slower than hourly at equal variable count, so the cap keeps a worst-case chunk solve near ~3s). This split is a **soft** terminal-neutral reset exactly like any other run boundary: SoC still carries across it via end-of-chunk seeding, but because each chunk is terminal-neutral the within-chunk carry benefit is what matters, and the chunked total can be slightly below a single full-horizon solve while staying ≥ per-day reset. The split is contiguous — no day is dropped. Guards: `test_group_clean_runs_splits_long_run_at_interval_cap` (chunking + full coverage) and `test_chunked_continuous_run_keeps_within_chunk_carry` (within-chunk carry survives a forced split). `carry_soc=False` (or single-day `len(dates) < 2`, or DA+ID with no intraday loaded) keeps the legacy per-day terminal-neutral loop and reports `carry_mode = "per_day_reset"`. `batch.attrs["da_id_carry_soc_supported"]` stays `True` (continuous DA+ID is implemented). The cockpit also renders a rolling 7-day revenue+FCE curve (≥3 days) and a weekday × ISO-week revenue heatmap (≥7 days) on top of the daily KPI bars to expose weekday/seasonal structure single-day KPIs hide. **Regression guards**: `test_replay_batch_carry_soc_produces_higher_revenue_than_reset` (DA-only) and `test_da_id_continuous_horizon_beats_per_day_reset` (DA+ID) construct overnight-arbitrage price paths where a no-op carry-over (e.g. only mutating `soc_init_frac` between independent terminal-neutral daily solves) would tie revenue with the reset mode and fail; `test_da_id_continuous_total_dominates_independent_days` asserts the global-≥-local property.
- **Intraday uplift (P5-A Phase 1)** (`analytics.calculate_intraday_uplift`): joins DA and IDA1 prices on the delivery timestamp and reports `avg|IDA-DA|`, `mean_signed` (positive ⇒ IDA usually prints above DA), and an annualised rebid uplift = `avg|IDA-DA| * rebid_share * duration_hours * cycles_per_day * capture_rate * 365.25`. The default `rebid_share=0.25` is intentionally conservative; EU practitioner ranges are typically 0.10–0.40 depending on DA commitment strategy and SoC management. The model is a single-stage screening estimate, NOT a proper two-stage DA + ID dispatch — it ignores rebid directionality (a BESS short on DA can only buy back in ID) and treats every absolute spread as fully captureable on average. IDA1 fetches are persisted to SQLite (`ida_prices_{zone}_seq{n}` tables) so browser refresh does not re-hit the ENTSO-E API; the Revenue Estimation UI tries the cache first and falls back to a button-gated live fetch. **Phase 2** would solve a two-stage stochastic dispatch problem: stage 1 commits a DA position under an IDA price forecast, stage 2 rebids into IDA against realised prices with explicit rebid-direction feasibility against the existing DA position. Implementation candidates: scenario-based stochastic MILP, or a sequential LP heuristic that re-optimises against the actual IDA print each day.
- **Sequential DA+ID policy (cockpit Phase 7)** (`dispatch.solve_sequential_da_id_dispatch`, `ida_forecast.build_ida_forecast`, `simulation.simulate_sequential_da_id_batch`): a sequential-heuristic realisation of the Phase-2 idea, surfaced in the Simulation Cockpit's "Forecast-driven IDA policy" expander. The solver runs Stage 1 (commit DA position via the DA-only MILP), Stage 2 (re-dispatch optimally against an IDA *forecast*, not the realised print), then settles the executed schedule at the *realised* IDA price with the same implicit-MtM accounting as the ex-post solver. It returns a three-way decomposition — `da_only_revenue_eur` (baseline), `realised_total_eur` (forecast-driven), `ceiling_total_eur` (= `solve_daily_da_id_dispatch` perfect-foresight upper bound) — with `forecast_error_cost_eur = ceiling - realised >= 0` and `captured_uplift_eur = realised - da_only` (which **can be negative** when the forecast misleads the rebid into churn losses; do NOT clamp it). Identity `captured + forecast_error == ceiling - da_only` always holds. The forecast is a hour-of-day (default, robust on short windows) or hour-of-week (weekday-aware, needs several weeks) climatology mean of the loaded IDA history. `build_ida_forecast`'s `forecast_mode` controls which history each day may see: `"loo"` (default) = leave-one-day-out cross-validation, every day EXCEPT the target — note this is **NOT walk-forward**, it may use future days from the loaded sample, so it is an unbiased *skill* estimate, not what a desk knew in real time; `"walk_forward"` = only days strictly before the target (drops the earliest day); `"in_sample"` = diagnostics only. It reports `coverage` / `n_buckets_filled` (history-backed buckets) / `n_buckets_requested` / `fallback_points` / `forecast_mode` so the UI labels forecast support honestly. **Framing caveat for any UI/doc copy**: arbitrage timing is driven mainly by price *order/shape*, but a forecast *level* error still changes whether a cycle clears (round-trip efficiency loss + VOM make a level shift flip the cycling decision), so do NOT claim "only shape matters". `simulate_sequential_da_id_batch` aggregates this per-day (standalone terminal-neutral, isolating forecast quality from multi-day SoC carry) and reports a guarded `capture_rate` (None when no rebid opportunity exists in the window). It uses **raw solver values** — the cockpit's sidebar DA-slippage capture haircut is intentionally NOT applied to this panel, which the UI states explicitly so the three-way numbers are not confused with the haircut-derated single-day/multi-day replays. **Risk gate / deadband** (`min_rebid_uplift_eur`, exposed in the panel as "Rebid deadband (EUR/MW/day)" × `power_mw`): the desk only rebids when the day's FORECAST-predicted uplift over holding the committed DA schedule exceeds the hurdle; otherwise it holds the DA schedule (`rebid=False`, executed schedule = DA schedule, settling to exactly `da_only`). The forecast uplift is non-negative by construction, and the gate is floored at `_REBID_UPLIFT_EPS_EUR=1e-9` so a zero/solver-noise uplift never registers as a rebid (deterministic `n_rebid_days`/`n_hold_days`); the default `0.0` therefore reproduces the always-rebid baseline (revenue-neutral) while a positive hurdle suppresses churn on marginal days where a thin forecast edge can flip into a realised loss. The batch summary reports `n_rebid_days` / `n_hold_days` and `min_rebid_uplift_eur`. **Forecast skill report** (`ida_forecast.compute_forecast_skill`, exposed via `summary["forecast_skill"]` and a cockpit metrics row + hour-of-day MAE chart): scores the climatology forecast against realised IDA in price space — `mae` / `bias` (signed forecast − realised) / `rmse` (EUR/MWh), `realised_std` for context, a per-local-hour MAE profile, and `skill_vs_da = 1 − MAE_forecast / MAE_da` vs the DA-as-IDA naive baseline (>0 ⇒ the climatology beats the "no rebid signal" null; ≤0 ⇒ the forecast-driven rebid rests on thin ice, so widen the deadband). This makes the deadband threshold an informed choice rather than a guess. A scenario-based stochastic MILP remains the heavier future upgrade.
- **Reserve price forecast skill (Phase 9.2b prep)** (`reserve_forecast.compute_reserve_forecast_skill`): scores a **block-of-day** climatology (6× 4h FCR/aFRR product blocks) of the loaded capacity-price history against realised, with a **flat sample-mean** naive baseline (no DA-as-IDA analog for reserve) — `mae`/`bias`/`rmse`/`realised_std`/`coverage` + `skill_vs_mean = 1 − MAE_clim/MAE_flat` (>0 ⇒ reserve prices carry block-of-day structure worth forecasting; ≤0 ⇒ a forecast-driven reserve commitment rests on thin ice) + a per-block MAE table. `leave-one-day-out` default (unbiased skill, NOT walk-forward), same `forecast_mode` semantics as `build_ida_forecast`. It is a forecastability **DIAGNOSTIC**, NOT a dispatch model (no activation energy). Surfaced in a cockpit expander gated on `ancillary.list_capacity_products()` (independent of IDA, unlike the forecast-policy panel). **Phase 9.2b** itself is now built and surfaced in the cockpit (see the Simulation Cockpit bullet's sixth row + forecast-effect gap panel). Locked v1 design: **reserve-first sequential** — reserve is committed FIRST (D-1 ~08:00, before the noon DA gate) under this price forecast, locking `reserve_mw` per 4h block; the DA position remains a full-power financial commitment, while physical IDA1 execution/rebid is capped by `power − reserve_mw`. So 9.2b ≤ the 9.2a perfect-foresight ceiling, the gap being the commitment-timing + forecast cost. Reserve auction order on D-1 (FCR ~08:00 → aFRR ~09:00 → DA 12:00 → IDA1 ~15:00) is why reserve-before-DA is the realistic ordering. A scenario-based stochastic MILP remains the heavier future upgrade.
